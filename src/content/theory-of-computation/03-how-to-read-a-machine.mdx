# How to Read a Machine

You now have the vocabulary, alphabets, strings, languages. thats what we learned previously, The next question is more practical: **how does a machine actually decide whether a string belongs to a language or not?**

> NOTE That if you found it hard to interpret this question, for example what a string is, go to the previous lesson

The answer is a state diagram. You'll meet it twice — first as a picture you read by hand, then as a 5-tuple of sets and functions that says exactly the same thing. The hand reading comes first because the notation is easier to trust once you've already understood the picture.

## The Anatomy of a State Diagram

A state diagram has four ingredients. That is all.

**States** — drawn as circles. Each circle is a snapshot of the machine's memory. A finite automaton has no tape, no counter, no variables — its entire memory at any moment is which circle it currently occupies.

**The initial state** — one state has an arrow pointing into it from nowhere. That is where the machine always starts, before reading any symbol.

**Accept states** — some states are drawn as double circles. If the machine finishes reading the entire input string while sitting in one of these, it answers *yes — this string is in the language*. If it ends anywhere else, the answer is *no*.

**Transitions** — arrows between states, each labeled with a symbol. When the machine reads a symbol from the input, it follows the arrow labeled with that symbol out of its current state. That is the entire computation.

<Callout tone="violet" label="The machine's only job">
  Read one symbol. Follow the matching arrow. Repeat until the string is exhausted. Check if the final state is an accept state. That is a finite automaton.
</Callout>

## The 5-Tuple Definition

Those four ingredients have a name. The machine they describe is called a **deterministic finite automaton**, abbreviated **DFA**. The name does real work — think of each word as a constraint.

- **Deterministic** — at every state, for every symbol in the alphabet, there is *exactly one* arrow leaving. No choice, no ambiguity, no surprises. Given the current state and the next input symbol, the next state is fully decided.
- **Finite** — the machine has a finite number of states. Its memory budget is exactly the number of circles in the diagram. Nothing else — no counter, no stack, no scratchpad.
- **Automaton** — an automatic machine. Once you hand it an input string, it runs on its own, symbol by symbol, with no human in the loop.

Put together: a machine with a bounded number of memory slots that reacts to each input symbol in exactly one way. That is the entire model. Every other automaton you meet later in the course is some relaxation of these three words.

Formally, a DFA is a 5-tuple:

$$
M = (Q,\ \Sigma,\ \delta,\ q_0,\ F)
$$

Each piece is one of the ingredients you just read about, written precisely:

- $Q$ — a finite set of **states**. The circles in the diagram.
- $\Sigma$ — a finite set of **input symbols**. The alphabet. Every arrow label is drawn from $\Sigma$.
- $\delta$ — the **transition function**, with signature $\delta: Q \times \Sigma \to Q$. Given a state and a symbol, it returns the next state. The arrows in the diagram *are* $\delta$, drawn graphically.
- $q_0$ — the **initial state**, with $q_0 \in Q$. The circle with the incoming arrow.
- $F$ — the set of **accept states**, with $F \subseteq Q$. The double-circle states.

The diagram and the tuple describe the same object. The diagram is the version humans read; the tuple is the version proofs are written about. Every DFA in this course can be written either way, and you should be able to translate between them on sight.

## A Parking Meter - A First Example

A parking meter that opens its gate when you have inserted $20 or more is a deterministic finite automaton in its core. The states track *amount paid so far*, the alphabet is *the coins you can insert*, and the accept state is *gate open*.

This meter accepts three coins, \$5, \$10, and \$25,
and opens at any total of \$20 or more.

This time, the states are:

- $q_0$ — $0 paid *(start)*
- $q_5$ — $5
- $q_{10}$ — $10
- $q_{15}$ — $15
- $q_\text{open}$ — $20 or more *(accept)* — once you land here, every further coin loops back, so the gate stays open

Try a few sequences in the input box above. Spaces are optional, so `$5 $10 $5` and `$5$10$5` parse identically:

- `$5 $10 $5` → $20 → **accepted**
- `$10 $25` → $35 → **accepted**
- `$5 $5` → $10 → **rejected** (gate stays closed)

This is what designing a DFA feels like when the language has an obvious memory structure. Once you decide *the running total is what I track*, every arrow in the diagram is forced — you cannot draw it wrong. You will not always be this lucky. The next section walks through a machine whose state meanings are not obvious; you will have to work them out by tracing strings.

## A Machine, Read by Hand

Here is a machine over the alphabet $\{a, b\}$. Step through it before reading the explanation below.

  alphabet={["a", "b"]}
  transitions={[

    { from: "q0", to: "q0", label: "b" },

    { from: "q1", to: "q2", label: "b" },
    { from: "q2", to: "q1", label: "a" },
    { from: "q2", to: "q0", label: "b" },
  ]}
  initial="q0"
  accepting={["q2"]}
/>

The same machine written as a **transition table** — each row says "from this state, on this symbol, go to this state":

| State | on $a$ | on $b$ |
| --- | --- | --- |
| $q_0$ — *start* | $q_1$ | $q_0$ |
| $q_1$ | $q_1$ | $q_2$ |
| $q_2$ — *accept* | $q_1$ | $q_0$ |

Picture and table contain the same six arrows. The picture shows you the shape; the table is what you scan when you're tracing by hand and don't want to lose your place.

Try the strings `ab`, `abb`, `b`, `bab`, `aab` before reading on. Form a guess about what language this machine accepts.



Let's trace each string in a table. Each row is one moment in time — the *Read* column is the symbol just consumed, and *State* is where the machine sits after consuming it.

Take the string $ab$:

| Step | Read | State |
| --- | --- | --- |
| — | (start) | $q_0$ |
| 1 | $a$ | $q_1$ |
| 2 | $b$ | $q_2$ |

End at $q_2$ — an accept state. **Accepted.**

Take the string $abb$:

| Step | Read | State |
| --- | --- | --- |
| — | (start) | $q_0$ |
| 1 | $a$ | $q_1$ |
| 2 | $b$ | $q_2$ |
| 3 | $b$ | $q_0$ |

End at $q_0$ — not an accept state. **Rejected.**

Take the string $b$:

| Step | Read | State |
| --- | --- | --- |
| — | (start) | $q_0$ |
| 1 | $b$ | $q_0$ |

End at $q_0$ — not an accept state. **Rejected.**

The pattern: **every accepted string ends in $ab$.** That is the rule.

Look back at the strings that accepted — $ab$, $aab$, $bab$ — each one's last two symbols are $a$ followed by $b$. The strings that rejected — $abb$, $b$ — do not. The machine is checking one thing and one thing only: *did the input just end in $ab$?*

In regex, this language is `/ab$/`. Every regex engine in every editor compiles a pattern like this into a state machine that runs in exactly the way you just traced — same circles, same arrows, same accept condition.

What matters here is not the specific language but that you can trace the machine mechanically, symbol by symbol, without guessing.

Written as a 5-tuple, the machine you just traced is:

$$
M = (\{q_0, q_1, q_2\},\ \{a, b\},\ \delta,\ q_0,\ \{q_2\})
$$

with $\delta$ given by the six entries on the arrows: $\delta(q_0, a) = q_1$, $\delta(q_0, b) = q_0$, $\delta(q_1, a) = q_1$, $\delta(q_1, b) = q_2$, $\delta(q_2, a) = q_1$, $\delta(q_2, b) = q_0$. The picture and the tuple are the same machine — pick the picture when you want intuition, the tuple when you want to prove something.

## What Each State Means

States are not arbitrary labels. Each state represents a **condition the machine needs to remember** in order to make future decisions.

$q_0$ means: *the last symbol I read was $b$, or I haven't read anything yet.* Not ready to accept — I would need to see an $a$ first.
$q_1$ means: *the last symbol I read was $a$.* One $b$ away from accepting.
$q_2$ means: *the last two symbols I read were $ab$.* This is the accepting condition.

Every time you design or read a state diagram, ask: *what does this state know?* The answer is always a condition on the input seen so far. That condition is the state's meaning. Once you read machines this way — as memory - designing them becomes a question of identifying what conditions matter to design what is required.

<Callout tone="orange" label="The finite memory constraint">
  A finite automaton has exactly as many memory slots as it has states. Three states means three possible conditions it can distinguish. If the language requires remembering something that grows with the input — like an exact count of $a$'s — a finite automaton cannot do it. You will prove this formally in Module 1. For now, just notice: the number of states is the machine's entire budget for memory.
</Callout>

## Two important notes

Before moving to Module 1, fix these two facts in your mental model:

**Every state must have exactly one outgoing arrow per symbol.** If the alphabet is $\{a, b\}$, every state needs an $a$-arrow and a $b$-arrow leaving it — no more, no less. A machine with a missing transition is incomplete. A machine where two $a$-arrows leave the same state is something different — in this case it would be a nondeterministic automaton, which Module 1 covers.

**The machine cannot look ahead.** It reads left to right, one symbol at a time, with no ability to peek at what comes next or go back. Every decision is made with only the current state and the current symbol. This constraint is what makes the model both limited and useful — limited enough to analyze mathematically, useful enough to match real tools like regex engines.
