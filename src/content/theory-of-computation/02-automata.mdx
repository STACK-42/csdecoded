# Finite Automata → Regex Engines

A **finite automaton** is the simplest computer imaginable: a fixed set of states, a starting state, some accepting states, and a transition function that reads one symbol and moves you to the next state. No memory beyond *which state am I in right now*. That’s it.

It sounds like a toy. It is also the engine inside every text search you have ever performed.

## The mapping

A regular expression is just a human-readable way to describe a finite automaton. When you write `\d{3}-\d{4}`, the engine compiles that pattern into a state machine, then walks it across your input one character at a time. The “regex” feels magical because the underlying math is exactly powerful enough to do this — and exactly weak enough to do it in linear time.

<Callout tone="neon" label="The provable limit">
Finite automata cannot count. That’s why regex famously cannot match *balanced* parentheses. The moment you need a stack, you need a more powerful machine — a pushdown automaton — which is the next module.
</Callout>

## The senior-engineer move

Knowing automata changes how you read regex. When you see catastrophic backtracking blow up production (the classic ReDoS attack), you can explain *why*: someone wrote a pattern that the engine had to simulate as an NFA with exponential paths. The fix isn’t guessing — it’s rewriting toward a pattern the engine can compile to a deterministic DFA.

That’s the difference between using a tool and understanding one.
